# 财务预测使用说明

## 这套工具解决什么问题

Forecast Lab 用历史财务数据做滚动回测，回答一个具体问题：**某项财务指标是否可以稳定预测，以及哪种方法在过去表现更可靠。**

它不是预算审批系统，也不会自动替代财务判断。系统给出的是基于历史数据和已配置业务假设的预测、误差与风险提示；预算目标、经营计划和最终口径仍由业务与财务共同确认。

一个实验完成后，用户应能够得到四类结果：

1. 未来若干期的金额预测及 P10-P90 区间。
2. 多个模型与业务基线的统一回测排行榜。
3. 各期间、各序列的偏差明细。
4. 已使用的协变量、节假日影响、增长率和手工情景假设。

## 一次完整实验怎么做

### 1. 准备数据

一行数据代表一个序列在一个期间的实际值。最低需要三个信息：

| 信息 | 作用 | 示例 |
|---|---|---|
| 时间字段 | 表示发生期间 | `2025-01`、`2025-01-31` |
| 目标字段 | 需要预测的金额或业务指标 | 营业收入、运营成本、补贴金额 |
| 序列维度 | 区分不同预测对象 | 组织、业务线、科目、产品 |

例如，按“业务线 + 财务科目”建序列后，`网约车 / 订单收入` 与 `网约车 / 运营成本` 是两条独立序列，应分别预测。

建议每条月度序列至少有 24 期历史数据，36 期以上更适合做稳定评测。日度和周度数据需要更长历史。

### 2. 完成字段映射和质量检查

在“字段与质量”步骤中配置：

- 时间字段、目标字段、序列维度字段。
- 影响因子与协变量候选字段。
- 静态属性，例如组织类型、城市覆盖数。
- 重复记录处理方式和目标缺失处理方式。

先看质量检查结果。以下情况应先处理再训练：

- 时间无法解析。
- 目标值为空或不是数值。
- 同一序列、同一期间存在重复记录且未明确聚合方式。
- 单条序列历史长度过短。

### 3. 设置预测范围和回测

| 配置 | 财务含义 | 建议 |
|---|---|---|
| 预测周期 | 希望往后看多少期 | 月度可先用 3-6 期 |
| 回测窗口 | 用历史多少个时点反复模拟“当时预测未来” | 先用 3 个窗口 |
| 训练模式 | 模型搜索时间预算 | 首次使用“快速验证”，正式比较用“标准评测” |

回测不是拿最后一期数据验证一次。系统会把历史切成多个时间窗口：每个窗口只使用当时可见的数据预测下一段，再与实际发生值比较。这样可以避免用未来信息判断模型好坏。

### 4. 配置预测驱动因素

“预测驱动配置”用于维护财务人员知道、但目标金额本身不能完整表达的信息。配置分为四类：

| 类型 | 什么时候使用 | 系统如何处理 |
|---|---|---|
| 协变量 | 数据里已有计划订单量、工作日数、燃油价格等字段 | 支持的模型把它们作为输入特征 |
| 增长率 | 已明确整体按期增长或下降的业务假设 | 对全部候选模型预测按每期复利调整 |
| 节假日/事件影响 | 五一、国庆、行业大促、政策窗口会影响指定月份 | 对匹配月份的回测和未来预测按影响率调整 |
| 手工情景协变量 | 没有计划订单数、燃油指数等源字段，但需要维护假设 | 生成可导出的未来假设，并按手工影响系数调整预测 |

#### 协变量：数据已经有字段时

协变量必须是无缺失的数值字段。需要选择：

- **已知未来**：预测期可以预先知道，例如计划订单量、工作日数、预算人数。回测时使用实际期间值；未来预测按“历史末值延续”或“历史均值延续”生成。
- **历史滞后**：只在发生后才能知道，例如已实现燃油价格、实际流量。当前由 XGBoost 作为滞后特征使用，不会被当作未来已知信息。

已知未来变量的未来值如果已有经营计划，应优先上传或维护计划值；“末值/均值延续”只是用于快速验证的默认假设，不应替代正式经营计划。

#### 五一、国庆等事件影响

月度数据可配置：

- 名称：`五一与国庆影响`
- 影响月份：`5`、`10`
- 影响率：例如收入 `+8%`、成本 `+5%`

影响率必须按预测对象维护。收入与成本对同一事件的方向和幅度通常不同，不应使用同一个系数覆盖所有科目。

日度或周度数据同样可以使用该配置：落在指定月份内的预测点会被调整。

#### 没有计划订单数、燃油指数时怎么办

没有历史字段时，模型无法从数据中学习该变量与目标金额的真实关系。此时不要把一个猜测值伪装成“协变量训练结果”，而应使用 **手工情景协变量**，维护三项内容：

| 配置 | 含义 | 计划订单数示例 | 燃油指数示例 |
|---|---|---:|---:|
| 基准值 | 当前情景起点 | 100 | 100 |
| 每期变化率 | 后续每期按何速度变化 | `+1.5%` | `+0.4%` |
| 目标影响系数 | 协变量变化 1%，目标变化多少 | `80%` | 成本可填 `35%` |

系统会生成“未来驱动假设”表，按每条序列、每个预测期间输出数值，并在 Excel 的 `未来驱动假设` Sheet 中保存。

影响系数需要业务确认：

- 计划订单数上升通常带动收入上升，可使用正系数。
- 燃油指数上升通常推高成本；若预测目标是收入，可能需要负系数或不配置。
- 手工情景是业务假设，不是模型从历史数据自动学习到的结论。

## 看什么指标

结果页首先看排行榜，再看回测窗口和序列明细。不要只看“谁排第一”。

### WAPE：主指标

`WAPE = 全部绝对误差之和 / 全部实际值绝对值之和`

它回答的是：**在整体金额规模上，预测偏差占实际金额多少。**

例如 WAPE 为 4%，可以理解为在当前评测样本中，每 100 元实际金额平均发生约 4 元绝对偏差。WAPE 越低越好，是模型排行榜的首要排序依据。

适合用于：收入、成本、费用等金额规模不同的多序列汇总比较。

### MAE：平均单期偏差金额

`MAE = 每个预测点绝对偏差的平均值`

MAE 用金额表示。它帮助回答：**单个月、单周或单日通常会偏多少钱。**

两个模型 WAPE 接近时，MAE 更低的模型通常更容易落地到预算滚动和经营跟踪。

### RMSE：大误差惩罚指标

RMSE 会对特别大的偏差施加更高惩罚。它适合识别“平均看起来还可以，但少数月份严重失真”的模型。

如果 WAPE 很低、RMSE 却明显偏高，应检查是否有大促、节假日、政策变化或数据异常没有配置。

### Bias Rate：系统性高估还是低估

`Bias Rate = 预测误差之和 / 实际金额绝对值之和`

- 正数：整体偏高估。
- 负数：整体偏低估。
- 接近 0：没有明显单边偏差。

财务使用时，低 Bias 很重要。持续高估会造成预算宽松或资源闲置；持续低估会造成费用、运力或现金安排偏紧。

### P10-P90 覆盖率：区间是否诚实

P10-P90 是预测区间，不是确定金额。覆盖率表示实际发生值有多少比例落在该区间内。

- 覆盖率很低：区间过窄，模型对不确定性估计不足。
- 覆盖率很高但区间很宽：区间可能过于保守，决策价值有限。

使用区间时，应同时看覆盖率和区间宽度。对现金、补贴、运力等风险敏感事项，优先关注 P90 或更保守的情景。

### 相对基线提升和回测窗口表现

系统始终保留 Last Value、Seasonal Naive、Rolling Mean 三类业务基线。

- **相对基线提升**：最佳模型相比最佳基线降低了多少 WAPE。
- **回测窗口表现**：最佳模型在多少个历史窗口优于最佳基线。

建议判断标准：

| 现象 | 建议 |
|---|---|
| 模型只在一个窗口领先 | 暂不作为正式预测依据 |
| 多数窗口领先且 Bias 接近 0 | 可进入业务试运行 |
| 个别序列 WAPE 高于 30% | 单独排查该组织、科目或产品 |
| 基线与复杂模型差异很小 | 优先选择更易解释、维护成本更低的基线 |

## XGBoost、AutoGluon、Prophet 怎么理解

### XGBoost：从历史规律和业务特征中找组合关系

可以把 XGBoost 理解为大量“如果……那么……”的组合判断。

它会同时观察：

- 最近几期的金额。
- 同比、环比附近的历史模式。
- 月份、季度、星期等日历信息。
- 已知未来协变量，例如计划订单量、工作日数。
- 历史滞后协变量，例如已实现燃油价格。

它擅长处理“订单量增长、工作日变化、季节性、最近趋势共同影响收入或成本”的非线性组合关系。

适合场景：

- 驱动因素较多。
- 数据量较充足。
- 关系不是简单直线，例如订单量超过某个水平后补贴成本明显变化。

注意事项：

- 需要较干净的数据和合理的协变量口径。
- 如果未来协变量是假设值，结果会随假设变化。
- 它能给出较好精度，但解释时应结合驱动因素和偏差明细，不应只引用一个模型分数。

### AutoGluon：自动组织多个时间序列方法并做比较

AutoGluon 不是一个单独的统计公式，而是一个自动化的模型评测与组合框架。

它会在给定时间预算内训练多个适合时间序列的候选方法，例如季节性方法、表格递归模型、指数平滑、Theta 模型等，再选择或组合表现更好的结果。

可以把它理解为：**先让多位分析员独立给出预测，再根据历史回测表现决定采用谁，必要时加权组合。**

适合场景：

- 还不确定哪种预测方法最适合当前科目。
- 希望快速获得一个稳健的“自动化基准”。
- 已知未来协变量需要与时间序列模型一起使用。

注意事项：

- 训练时间通常比单一模型更长。
- 输出仍需以统一 WAPE、Bias、窗口表现判断，不能直接使用内部原始分数。
- 当数据很短或业务规律变化很快时，自动组合不一定优于简单基线。

### Prophet：把趋势、季节性和已知驱动拆开看

Prophet 将一条时间序列拆成几部分：

1. 长期趋势：金额总体在上升、下降还是趋于平稳。
2. 季节性：例如每年春节、暑期、年末是否存在规律性变化。
3. 已知未来协变量：例如计划订单量、工作日数对金额的额外影响。

它的优势是业务沟通直观：可以解释“趋势贡献、季节贡献、协变量贡献”共同形成了预测值。

适合场景：

- 存在明显月度、周度或年度节奏。
- 需要向管理层解释趋势和季节性。
- 已知未来的业务计划较明确。

注意事项：

- 对突发结构变化的适应性有限。
- 如果节假日、活动影响没有持续历史规律，应使用事件影响配置或手工情景，而不是假设 Prophet 会自动知道。

### 不必把模型名称当成结论

选模型的顺序应是：

1. WAPE 是否优于基线。
2. Bias 是否可接受。
3. 是否在多个回测窗口稳定领先。
4. 关键序列是否存在严重偏差。
5. 使用的业务假设是否可解释、可维护。

模型名称只是实现方式。一个稳定的 Seasonal Naive 可能比不稳定的复杂模型更适合正式使用。

## 结果页怎么读

| 页面区域 | 应回答的问题 |
|---|---|
| 评测总览 | 最佳模型是否优于基线，提升是否稳定 |
| 趋势分析 | 预测曲线是否跟随实际变化，未来区间是否合理 |
| 偏差分析 | 哪些月份高估/低估最多，是否存在系统性方向偏差 |
| 模型榜单 | 模型之间 WAPE、MAE、Bias、覆盖率的权衡 |
| 数据与配置 | 本次使用了哪些字段、影响因子和手工情景假设 |

对于正式财务使用，建议先从“数据与配置”确认口径，再查看“模型榜单”和“回测窗口表现”，最后进入单条序列的趋势与偏差明细。

## 合成滴滴出行示例

项目提供了一份合成示例：`sample_data/didi_finance_forecast_example.xlsx`。该数据不代表滴滴实际财务信息。

示例可配置为：

- 序列维度：业务线、财务科目。
- 已知未来协变量：计划订单量、工作日数。
- 历史滞后协变量：燃油价格指数。
- 事件影响：五一、国庆影响 5 月与 10 月。
- 无源情景：手工维护计划订单数和燃油指数的基准、变化率与影响系数。

对应的实验结果和导出报告会保存在本机 `runtime/experiments/` 下。所有数据、模型与报告都保留在本地，不会上传到外部服务。
